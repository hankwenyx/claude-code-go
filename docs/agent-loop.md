# Agent Loop — 核心对话循环

Agent Loop 是 Claude Code 的"大脑主循环"。每当你输入一条消息，它负责完成从"接收输入"到"输出结果"的全部工作：调用模型、执行工具、处理错误、决定是否继续。

---

## 一句话理解

> **用户说话 → 模型思考 → 调用工具 → 回来继续思考 → 直到任务完成**

这个过程不是一次性的，而是一个循环（Loop）。模型可能调用十几次工具才完成一个任务，每次调用工具的结果都会反馈回来，让模型继续决策。

---

## 整体调用链

```
你的输入
  │
  ▼
handlePromptSubmit()          ← 处理提交，识别斜杠命令
  │
  ▼
processUserInput()            ← 规范化输入，构造消息
  │
  ▼
query()                       ← 公开入口，收集工具调用记录
  │
  ▼
queryLoop()  ◄── 真正的主循环（while true）
  │
  ▼
queryModelWithStreaming()      ← 调用 Claude API（流式 SSE）
  │
  ▼
anthropic.beta.messages.create({ stream: true })
```

## 核心循环图解

可以把 `queryLoop()` 想象成一台"思考机器"，它每轮做这几件事：

```
┌─────────────────────────────────────────────────────────────────────┐
│                         queryLoop()                                  │
│                                                                      │
│  ┌──────────┐    ┌──────────────┐    ┌──────────────────────────┐   │
│  │ 准备消息  │───▶│  调用模型    │───▶│   模型返回了什么？        │   │
│  │ 截断/压缩 │    │  流式 SSE   │    │                          │   │
│  └──────────┘    └──────────────┘    └──────┬───────────────────┘   │
│                                             │                        │
│                               ┌────────────┴─────────────┐          │
│                               │                          │          │
│                         只是文字回复                  包含工具调用    │
│                               │                          │          │
│                               ▼                          ▼          │
│                          检查 Stop Hooks           执行工具          │
│                               │                    （可并发）        │
│                               │                          │          │
│                           任务完成 ◄─────────────── 结果追加到消息   │
│                                                          │          │
│                                                    继续下一轮 ───────┘
└─────────────────────────────────────────────────────────────────────┘
```

## 每一轮都在做什么（详细版）

### 第 1 步：准备要发给模型的消息

模型不能看到无限长的对话历史，所以每次调用前需要"整理"消息：

```
原始消息历史
  │
  ├─ 1. 只取上次压缩边界之后的消息
  ├─ 2. 超大的工具结果截断（避免单个结果占满窗口）
  ├─ 3. 删除重复的工具结果（微压缩，不需要 AI 参与）
  └─ 4. 如果 token 超过阈值 → 触发自动压缩（见上下文管理文档）
```

### 第 2 步：流式调用模型

Claude API 是流式返回的（SSE），每个 content block 完成时立刻 yield 给上层。
关键点：**不是等整个回复完成才处理，而是边接收边处理。**

```
流式事件流：
  message_start       → 初始化消息对象
  content_block_start → 开始一个内容块（文字 / 工具调用 / 思维链）
  content_block_delta → 增量内容到来（文字追加、JSON 参数追加）
  content_block_stop  → 本块完成 → 立即 yield 给上层处理
  message_delta       → 写回最终 usage / stop_reason（直接修改对象引用）
  message_stop        → 整条消息结束
```

一个特别的优化：**工具可以在模型还没说完话时就开始执行！**

```
模型正在 streaming... （还没说完）
  ├─ tool_use block A 完成 → 立刻开始执行 A
  ├─ tool_use block B 完成 → 立刻开始执行 B（和 A 并发）
  └─ 模型说完 → 消费剩余工具结果
```

这就是 `StreamingToolExecutor` 做的事情，它让工具执行和模型输出互相重叠，节省等待时间。

### 第 3 步：决定接下来怎么办

模型的回复分两种情况：

**情况 A：只有文字，没有工具调用**
```
没有工具 → 检查是否有错误（413 / max_tokens）→ 运行 Stop Hooks → 任务完成
```

**情况 B：包含工具调用**
```
有工具调用 → 执行工具 → 把结果追加到消息历史 → 继续下一轮循环
```

### 第 4 步：执行工具

工具不是随意并发的，有严格的调度规则：

```
只读工具（Read / Grep / Glob）  ─── 可以互相并发执行
写入工具（Bash / Edit / Write） ─── 必须串行，独占执行

Bash 出错时 → 自动取消同批次的其他工具（因为它们可能有依赖）
```

### 第 5 步：循环还是结束？

每轮结束后判断：
- 有工具结果 → 继续循环（让模型看到结果）
- 没有工具 + 一切正常 → **完成**
- 遇到不可恢复的错误 → **终止**

## 什么时候会"自我恢复"？

Loop 内置了几种自动恢复机制，不需要用户干预：

| 问题 | 自动恢复方式 |
|------|------------|
| 模型输出太多（max_tokens） | 先尝试升级输出上限到 64k，不行再注入"请继续"（最多 3 次） |
| 消息太长（413 错误） | 先压缩上下文，再重试 |
| 模型不可用（529 过载） | 指数退避重试；严重时切换到备用模型 |
| Stop Hook 检测到问题 | 把 hook 的反馈注入对话，让模型修正后再交付 |

---

## 终止方式一览

Loop 有多种退出方式，每种都有明确的原因：

```
正常退出：
  completed         → 模型完成任务，无工具调用，Stop Hooks 通过

用户触发：
  aborted_streaming → 用户在模型输出时按 Ctrl+C
  aborted_tools     → 用户在工具执行时按 Ctrl+C

错误退出：
  model_error       → API 异常且无法恢复
  prompt_too_long   → 压缩后仍然超出窗口
  blocking_limit    → Token 阻塞上限（auto-compact 关闭时）

限制退出：
  max_turns         → 达到最大轮数上限
  hook_stopped      → Hook 强制停止
  stop_hook_prevented → Stop Hook 阻止继续
```

## 错误如何传递给模型？

工具执行出错时，错误会被包装成一个特殊的工具结果返回给模型，**循环不会停止**——模型可以看到错误并决定如何处理：

```
tool.call() 抛出异常
  │
  ▼
<tool_use_error>Error calling tool: {message}</tool_use_error>
  │
  ▼
作为 tool_result 追加到消息历史
  │
  ▼
模型在下一轮看到错误，决定：重试 / 换方法 / 告诉用户
```

只有真正无法恢复的情况（API 崩溃、用户中断）才会终止整个 Loop。

## 消息类型速查

Loop 内部有多种消息类型，但并非所有消息都会发给 API：

| 类型 | 会发给 API？ | 用途 |
|------|------------|------|
| `user` | ✓ | 用户输入 / 工具结果 |
| `assistant` | ✓ | 模型回复（含工具调用、思维链） |
| `system compact_boundary` | ✓（作为 system） | 标记压缩发生的位置 |
| `attachment` | ✗（转为 user） | 文件附件、记忆注入等 |
| `progress` | ✗ | 工具执行中的进度更新（仅 UI 用） |
| `tombstone` | ✗ | 模型切换时废弃的消息标记 |
