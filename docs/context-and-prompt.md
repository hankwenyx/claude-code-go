# 上下文管理与 Prompt 系统

"上下文管理"回答的是一个核心问题：**每次调用模型时，到底把什么内容发过去？**

这包括系统提示词的构建、对话历史的取舍、Token 预算的分配，以及如何在不破坏缓存的情况下动态更新内容。

---

## 一张图看懂系统 Prompt 的构成

发给模型的内容由三部分组成：

```
┌─────────────────────────────────────────────────────────────┐
│                      API 请求内容                            │
│                                                             │
│  ┌─────────────────────────────────────┐                    │
│  │          System Prompt              │                    │
│  │                                     │                    │
│  │  ① 静态区（全局缓存）                │                    │
│  │    - 角色定义 / 行为规范              │                    │
│  │    - 工具使用指南                    │                    │
│  │    - 输出风格要求                    │                    │
│  │  ─────────────── 分割线 ──────────  │                    │
│  │  ② 动态区（每轮更新）                │                    │
│  │    - CLAUDE.md 记忆内容              │                    │
│  │    - 当前环境信息（cwd/git/OS）       │                    │
│  │    - MCP 服务器指令                  │                    │
│  │    - 语言偏好 / 输出风格设置          │                    │
│  └─────────────────────────────────────┘                    │
│                                                             │
│  ┌─────────────────────────────────────┐                    │
│  │     Messages（对话历史）             │                    │
│  │    user → assistant → user → ...   │                    │
│  └─────────────────────────────────────┘                    │
│                                                             │
│  ┌─────────────────────────────────────┐                    │
│  │     Tools（工具定义列表）             │                    │
│  └─────────────────────────────────────┘                    │
└─────────────────────────────────────────────────────────────┘
```

**为什么分静态区和动态区？**
因为 Anthropic 的 Prompt Cache 是基于内容前缀缓存的。静态区内容不变，可以全局缓存；动态区内容会变（比如 MCP 工具列表），放到分割线后面，避免动态内容影响前面稳定内容的缓存命中。

## CLAUDE.md 是怎么注入的

CLAUDE.md 文件是用户自定义指令的主要载体。Claude Code 会从多个位置加载它们，**越靠近你的项目，优先级越高**：

```
/etc/claude-code/CLAUDE.md        ← 企业管理员统一设置（最低优先级）
~/.claude/CLAUDE.md               ← 你的全局个人偏好
  /project-root/CLAUDE.md         ← 项目级规则（提交到 git）
    /project-root/.claude/rules/  ← 细粒度规则（可按文件路径过滤）
      /project-root/CLAUDE.local.md  ← 本地覆盖（不提交，gitignore）
```

加载时从根目录一路向 cwd 加载，靠近项目的文件后加载，优先级更高。文件内还支持 `@include` 引用其他文件（最多嵌套 5 层）。

注入时会在前面加上提示语：
> "以下是代码库和用户指令，IMPORTANT: 这些指令会覆盖默认行为，你必须严格遵守。"

## 上下文窗口与 Token 预算

### 模型能"记住"多少？

```
默认上下文窗口：200,000 tokens（约 15 万汉字）
超大模型/实验：1,000,000 tokens

有效窗口 = 总窗口 - 20,000（保留给压缩摘要输出）
```

### 触发自动压缩的阈值

```
有效窗口
│
├─ 超过 87%（约 163,400 tokens）→ 触发自动压缩
├─ 超过 90%（约 170,000 tokens）→ 展示警告
└─ 超过 97%（约 184,000 tokens）→ 展示错误，手动 /compact 被阻断
```

### 模型每次最多输出多少？

Claude Code 默认不会预留太多输出 Token，因为大多数回复不需要那么多：

```
默认请求值：8,000 tokens（够用的 p99）
           ↓ 如果模型说"我还没说完"（max_tokens 截断）
自动升级到：64,000 tokens
           ↓ 如果还不够
最多重试：  3 次，每次注入"请继续"

模型上限（仅供参考）：
  opus-4-6 / sonnet-4-6 → 128,000 tokens
  其他 claude-4 系列    → 64,000 tokens
  claude-3 系列         → 8,192 tokens
```

## 对话压缩：太长了怎么办？

当对话历史太长，压缩系统会用 AI 生成一份"会议纪要"，替换掉旧的对话记录，同时保留关键信息。

### 压缩触发 → 执行的完整流程

```
检测到 token 超过阈值
       │
       ▼
先试：会话记忆剪枝（无需 AI，直接删除旧记录）
       │ 不够用
       ▼
正式压缩：
  1. 执行用户配置的 pre_compact hook（shell 命令）
  2. 构建压缩提示词
  3. 调用 AI（max 20,000 tokens 输出）生成摘要
  4. 如果摘要请求本身太长（413）→ 删掉最旧的对话再重试（最多 3 次）
  5. 解析摘要（9 个固定章节的结构化格式）
  6. 恢复最近读取的 5 个文件（保持文件上下文连续性）
  7. 重新广播工具列表 / MCP 指令
  8. 执行 SessionStart hook
       │
       ▼
对话历史被替换为：
  [COMPACT_BOUNDARY 标记]
  [摘要消息]
  [最近 5 个文件附件]
```

### AI 生成的摘要包含 9 个章节

| # | 章节 | 用途 |
|---|------|------|
| 1 | Primary Request and Intent | 你让 Claude 做什么 |
| 2 | Key Technical Concepts | 涉及的技术概念 |
| 3 | Files and Code Sections | 操作过的文件 |
| 4 | Errors and Fixes | 遇到的问题和解法 |
| 5 | Problem Solving | 解题思路 |
| 6 | All User Messages | 用户说过的话摘要 |
| 7 | Pending Tasks | 未完成的任务 |
| 8 | Current Work | 当前正在做什么 |
| 9 | Optional Next Step | 建议下一步 |

### 各种压缩策略

不是所有情况都用 AI 压缩——能不花钱就不花钱：

```
微压缩（免费）    → 删除重复的工具结果，无 AI 调用
会话记忆剪枝（免费）→ 删除旧会话笔记，无 AI 调用
自动压缩（收费）  → 超阈值时自动触发，AI 生成摘要
响应式压缩（收费）→ 收到 413 错误时紧急触发
手动压缩（收费）  → 用户主动运行 /compact
```

## Prompt Cache：省钱的关键

反复调用时，如果系统提示词没有变化，Anthropic 会缓存它，你不需要每次都重新计算 Token 费用。

### 缓存分层策略

```
系统提示词分块（从前到后）：

Block 1: 计费归因 header         → 不缓存（每次不同）
Block 2: CLI 前缀信息            → 不缓存
Block 3: 静态规则（boundary 前）  → 全局缓存（最稳定，跨组织共享）
Block 4: 动态内容（boundary 后）  → 不缓存（会变化）
```

```
消息历史（cache_control 标记位置）：

[...历史消息...]  ← 不标记
[倒数第2条用户消息]  ← 不标记
[最后一条用户消息]  ← 最后一个 block 打上 cache_control: ephemeral
```

### 1 小时长效缓存

默认缓存 TTL 是 5 分钟。满足条件时升级到 1 小时：
- 是 claude.ai 订阅用户且未超额
- 是特定类型的主线程查询

**为什么要"锁定"缓存条件？**

如果会话中途切换了某个开关（比如开启了 fast_mode），系统提示词的内容就变了，之前的缓存就废了。为了避免这种"缓存失效"，这些开关一旦在 session 开始时确定，就不再改变——即使后来关闭了，也继续发送同样的 header，直到本次 session 结束。

## 思维链（Extended Thinking）

### 哪些模型支持？

```
opus-4-6 / sonnet-4-6  →  Adaptive Thinking（无预算上限，模型自主决定）
opus-4 / sonnet-4 / haiku-4  →  普通 thinking（需要设置 budget_tokens）
claude-3-*  →  不支持
```

### Thinking Clear 机制

如果距离上次 API 调用超过了 1 小时（缓存过期），发送时会附带清除旧 thinking 块的参数，防止报错 "thinking blocks cannot be modified"。

### Ultrathink

在提示词中写 `ultrathink`，会触发高强度思维模式（注入 `ultrathink_effort: high` 附件）。

## 附件处理

### @ 提及文件时发生了什么

```
用户输入 "@src/utils/foo.ts"
       │
       ├─ 检查是否有 deny 规则
       ├─ 检查文件大小
       ├─ 是 PDF 且页数很多？→ 只发"这是个 PDF，共 X 页，大小 Y"（不发内容）
       ├─ 文件已在上下文且没改过？→ 告诉模型"已经有了，不重复发送"
       └─ 正常情况 → 读取文件内容，附加到消息中
                      └─ 超出 token 限制？→ 不发（at-mention 模式）
```

### 图片处理

粘贴图片时：
1. 转为 base64 + 自动降采样（太大的图会被压缩）
2. 单图 token 上限约 2,000 tokens
3. 整个请求最多 100 个媒体块，超出时删最旧的

### 相关记忆自动注入

每轮对话时，系统会异步预取与当前任务相关的记忆文件，悄悄注入到消息中（包在 `<system-reminder>` 标签里，不破坏缓存）：
- 单轮最多 5 个文件，总量 20KB
- 整个 session 累计上限 60KB
